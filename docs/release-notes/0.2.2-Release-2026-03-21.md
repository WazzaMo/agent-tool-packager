# Summary

This is the first formal release note for Agent Tool Packager (ATP). It reflects the three
product features described under `docs/features/`, the planning notes that shaped Station
layout, catalog rules, configuration accuracy, and Safehouse initialization behavior, and the
closed items from the install-process todo note dated 2026-03-15.

The published CLI version matches `package.json` (this document: 0.2.2).

# Feature 1 — Catalog, Station, and installation

ATP provides a Station (default `~/.atp_station`, overridable with `STATION_PATH`) that
holds configuration, the package catalog, and installed package metadata. Users can
initialize the Station, list the catalog, and install packages into a project
Safehouse (`.atp_safehouse`) or the Station, with separate defaults for prompt material
and binaries (`--project` / `--station`, `--user-bin` / `--project-bin`).

Package types include Rule, Skill, MCP, Command, and Experimental. The CLI supports agent
nomination and handover, listing installed packages at Station and Safehouse scope,
and removing packages (including Station removal with optional exfiltrate for
migration-style workflows).

Planning context: package metadata and catalog shape are described in
`docs/notes/2026-02-25-plan-package-metadata-and-catalog.md` and the feature doc
`docs/features/1-package-definition-and-installation.md`.

# Feature 2 — Package developer support

Developers can author packages incrementally: skeleton creation, metadata commands
(name, type, version, license, copyright, usage, developer), staging file components
and executable bundles (including UNIX-style layouts or `--exec-filter` for non-conformant
trees), validation and summary, and adding a validated package to the user
section of the Station catalog (producing `package.tar.gz` alongside `atp-package.yaml`).

Feature detail and acceptance-style expectations: `docs/features/2-package-developer-support.md`.
Integration testing direction appears in `docs/notes/2026-03-14-test-package-developer-integration-test.md`.

# Feature 3 — Install process (project / Safehouse)

End-to-end project workflow: resolve the project base (`.git` / `.vscode` within a short
parent walk, or `SAFEHOUSE_PROJECT_PATH`), initialize the Safehouse, select an agent
from Station `agent-paths`, install from the catalog, unpack bundles per scope, place
prompt files under the agent’s configured paths, patch prompt content where paths must reflect
binary location, and record the install in `manifest.yaml`.

Feature narrative and step breakdown: `docs/features/3-package-install-process.md`.

# Package install process — work closed from project todo

The note `docs/notes/2026-03-15-todo-questions-on-package-install-process.md` tracked gaps between
Feature 3 and the implementation. The items below were completed (status as of 2026-03-18 in that
note, plus home-directory hardening documented elsewhere for `isHomeDirectory` / init).

### Workflow and agent commands

`atp agent` requires an existing Safehouse (init first). Feature 1 examples were updated to show
`atp safehouse init` before `atp agent`. Only agent names present in the Station `agent-paths`
configuration are accepted for `atp agent <name>` and `atp agent handover to <name>`; unknown
names exit with a clear error. Integration tests cover this behavior.

### Manifest and Station registration

The Safehouse manifest matches the Feature 3 shape: a `Safehouse-Manifest` wrapper and per-package
fields `name`, `version`, `source` (station or local), and `binary_scope` (user-bin or
project-bin). The loader still accepts a legacy flat layout for compatibility.

When the Station exists, `atp safehouse init` appends the new Safehouse path to
`atp-safehouse-list.yaml` under `STATION_PATH`.

### Install mechanics

Prompt files use `{bundle_name}` placeholders (per Feature 3); during install, those tokens are
replaced with the installed bundle path when skills and rules are copied.

Program assets from the package are copied to the binary install location: user scope uses
`~/.local/bin`, project scope uses `.atp_safehouse/{package}-exec/bin`. UNIX-conformant versus
non-conformant bundle expectations are spelled out in Feature 3; catalog add (Feature 2) is
responsible for producing the staged layout the installer consumes.

### Dependencies

Install honors explicit dependency pre-approval: without `--dependencies`, a missing dependency fails
with a clear error; the user stays in control (documented in Feature 1).

### Project base discovery

`findProjectBase` walks upward from the current directory (limited radius) for `.git` or `.vscode`,
or uses `SAFEHOUSE_PROJECT_PATH` when set. It is used by `atp agent`, `atp safehouse init`,
`atp safehouse list`, `atp install`, and `atp remove safehouse` so commands work from
subdirectories and resolve the correct Safehouse root. Home-directory detection for init was
added for the anti-pattern case and later hardened (see Safety and configuration alignment).

### Code documentation (Feature 3 touchpoints)

JSDoc and import layout were brought in line with `docs/clean-code.md` on the main Feature 3
sources (`install.ts`, `copy-assets.ts`, `agent.ts`, paths/load/init helpers, and related modules).
The `copy-assets` module header was updated to describe tarball handling and binary scopes.

### Outstanding clarifications from the same note

The todo note still records low-severity follow-ups: no interactive stdin confirmation when the
project base is uncertain (the CLI exits non-zero and points at `SAFEHOUSE_PROJECT_PATH` instead);
possible rename of `cwd` parameters to `projectBase` in some APIs for clarity; and a documented
distinction that install copies from catalog-prepared package material rather than re-deriving
UNIX layout at install time.

# Safety and configuration alignment

- Safehouse init and `$HOME`: Init refuses to create `.atp_safehouse` when the resolved
  project root is the user’s home directory (including when markers or `SAFEHOUSE_PROJECT_PATH`
  point at home). Override only with `ATP_ALLOW_HOME_SAFEHOUSE=1` (exactly `1`). See
  `docs/notes/2026-03-21-plan-safehouse-init-home-hardening.md`, `README.md`, and
  `docs/configuration.md`.

- Station files and catalog schema: Configuration docs and implementation were aligned
  (for example catalog `packages.standard` / `packages.user` object lists, and Station
  safehouse list naming under the `atp-` file convention). See
  `docs/notes/2026-03-19-plan-config-doc-alignment.md` and `docs/configuration.md`.
