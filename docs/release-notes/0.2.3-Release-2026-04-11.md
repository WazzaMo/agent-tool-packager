# Summary

Version 0.2.3 centres on two themes. First, one catalog entry can ship several agent-facing
shapes (rule text, skills, MCP servers, hooks, and so on) as separate parts inside one
`atp-package.yaml`, instead of splitting them across dependent packages. Staging and the
published tarball use a per-part prefix inside `stage.tar` so files from different parts
stay distinct. Authors who still want the older single-type flow can opt in with
`atp create package … --legacy`. Bundle authoring also accepts **`--skip-exec`** when a
bundle directory has no executables to register (see the authoring subsection below).
Bundle **source directories** may also live **outside** the package tree (for example a
sibling folder or **`../…`** paths), using the **same path resolution rules as components**;
the manifest stores a **relative** `path` when the bundle is under the package root, and
the directory **basename** when it is not, and **`stage.tar`** is updated accordingly for
both **legacy root** bundles and **Multi** **`part … bundle add`** flows.

Second, each supported agent gets an installer policy that maps part kinds to real
directories and config files, merges MCP and hook payloads where the product uses JSON or
TOML, and shares skill layout logic where several agents follow the same Agent Skills
pattern. Install can refuse ambiguous merges until the operator passes force or skip
flags, and stderr messages call out the concrete file and the remedy.

## Package installation requires a configured Safehouse

In addition, catalog **`atp install`** (including **`--station`**) and
**`atp remove safehouse`** require an explicit Safehouse **`agent`** in
**`.atp_safehouse/atp-config.yaml`**. After **`atp safehouse init`**, run
**`atp agent <name>`** before installing or removing packages from the Safehouse. ATP no
longer treats a **null** or blank **`agent`** as **cursor**; it exits **non-zero** with a
short message pointing at **`atp agent`**.

So the user is required to initialise the Safehouse in their project folder and set the
agent-in-use for that project, before attempting to install a package.

```bash
atp safehouse init
atp agent codex
```

**`atp catalog remove <package-name>`** removes a **user** package from the Station,
**`atp-catalog.yaml`** (**`packages.user`**) and deletes **`user_packages/<package-name>/`**
when that directory exists. It does **not** remove **`packages.standard`** entries; edit
**`atp-catalog.yaml`** or your standard catalog source for those.

By default, removal **refuses** (exit **1**) when **`atp-safehouse-list.yaml`** still points
at a Safehouse whose **`manifest.yaml`** lists that package, so authors do not drop the
catalog row while installs remain registered. Use **`--from-catalog-only`** to remove only
the Station catalog material (previous behaviour; may orphan installs), or
**`--and-from-projects`** to run **`atp remove safehouse`** semantics for each matching
registered project first, then remove the catalog row. The two flags are mutually
exclusive.

# CLI self-version check

New versions of ATP will be published and it's useful to know when a new version is available.
Now ATP will indicate if a new version is on NPMJS registry and provides a command option
to find out latest versus current versions.

After argv parse, a root **`preAction`** hook may run **`maybePrintUpdateNotice`**: one
short **`fetch`** to the registry (default **3s** timeout), then a single **stderr** line
when **`semver`** reports the published **`latest`** is strictly newer than
**`atp_version()`**. The check is **skipped** when argv includes **`--help`**, **`-h`**,
**`--version`**, or **`-V`**; when **`ATP_SKIP_UPDATE_CHECK`** is **`1`** or **`true`**;
for dev builds (**`0.0.0-dev`** or any version ending in **`-dev`**); when the Station
directory does not exist (no throttle file location); when **stdout** is not a TTY unless
**`ATP_UPDATE_CHECK=verbose`**; when the throttle file **`${STATION_PATH}/.atp_last_registry_check`**
shows a successful check within the last **24 hours**; when the registry base URL is
invalid; or when the registry read fails (no error is thrown for the user’s command).

- **Registry URL:** optional **`configuration.npm-registry-base-url`** in Station
  **`atp-config.yaml`** must be an **HTTPS** origin (mirrors); if omitted, the public npm
  registry is used. Documented in **`docs/configuration.md`**.

- **`atp --latest`:** handled before Commander parse when argv is standalone **`--latest`**
  (no subcommand). Prints **Current** / **Latest** on **stdout** when the read succeeds;
  stderr for skip, bad config, or fetch failure; always exits **0**.

- **Notes:** **`docs/notes/2026-04-17-note-atp-update-notice-code.md`**,
  **`docs/notes/2026-04-17-note-atp-update-notice-tests.md`**, and the plan
  **`docs/notes/2026-04-16-plan-atp-self-version-check-npm.md`**.

# Multi-type packages

## Intent and compatibility

Multi-type packaging exists so one distributable can combine material that used to require
multiple packages (for example a skill plus an MCP server that belong together). The
catalog still names one package; `atp install` installs every part in list order. There
is no automated conversion from legacy single-type manifests to Multi: authors who need
the new shape start from a new Multi skeleton and move content manually.

Legacy layout keeps a single root `type` and root-level `usage`, `components`, and
optional `bundles`, with a flat `stage.tar`. Multi layout uses root `type: Multi` (default
on new skeletons), a `parts` array, and no root-level usage or components when authoring in
the modern flow. Root `atp package usage` and `add usage` are not used for Multi; each
part carries its own usage lines.

## Authoring and staging

- `atp package newpart <keyword>` and `atp package part add <type>` append parts.
  Two or more parts may share the same **`type`**; validation succeeds without stderr
  warnings for that case alone.

- Per-part commands include `part <n> usage`, `part <n> add usage`,
  `part <n> component …`, `part <n> bundle add …`, `part <n> remove`,
  `part <n> component remove`, and `part <n> bundle remove`. These apply only while
  editing the unpublished package directory. They do not apply to packages already in
  the catalog or copied into a Safehouse.

- Multi packages stage paths under `part_{index}_{Type}/…` inside `stage.tar`. Legacy
  packages keep paths at the tarball root as before.

- `atp package type` can select `multi` or a single root type where the manifest rules
  allow it; incompatible changes are blocked when `parts` is already populated.

- Bundle directory basenames must be unique across the whole package (collision exits 2).
  Re-adding the same bundle path for the same part stays idempotent (exit 0).

- Component basenames must be unique across the whole package for Multi, at add time and
  in validate.

### Data-only bundles (`--skip-exec`)

Previously, every bundle had to expose executables either via a UNIX-style `bin/`
directory or an explicit `--exec-filter` glob so catalog enrichment could register
`program` assets. The new **`--skip-exec`** flag opts out of that: the directory is still
staged into `stage.tar` and published with the package, but the manifest records
`skip-exec: true` on the bundle row and **no** `exec-filter`, and the catalog `assets` list
does not add `program` rows for that tree (useful for MCP bundles that ship only config or
other non-binary payloads). **`--skip-exec`** cannot be combined with **`--exec-filter`**.

Examples:

- Multi part, Mcp part index 2, config-only folder `mcp-assets/` under the package root:
  `atp package part 2 bundle add mcp-assets --skip-exec`

- Legacy single-type package (root `bundles`), same idea:
  `atp package bundle add support-files --skip-exec`

After catalog add, installs still receive the staged files; only executable discovery is
skipped for bundles marked this way.

Implementation and test pointers:
**`docs/notes/2026-04-12-coding-bundle-skip-exec-authoring.md`**,
**`docs/notes/2026-04-12-test-bundle-skip-exec-authoring.md`**.

### Skill bundles and catalog `assets`

On `atp catalog add package`, enrichment walks each **Skill** part’s bundle directories
under `part_N_Type/…` (and root **Skill** package **`bundles`**) and writes `type: skill`
rows for the files there, so layouts such as `skill.yaml` plus `skill.md` or nested
`scripts/` reach install without listing every path under **`components`**; **`skip-exec`**
still omits only **`program`** rows for that bundle, not skill discovery. Republish with
**`atp catalog add package`** to refresh **`assets`** on existing station copies, and keep
one logical skill tree per **Skill** part so install-time assembly rules still apply. Plan
and test notes:
**`docs/notes/2026-04-12-plan-catalog-skill-bundle-asset-enrichment.md`**,
**`docs/notes/2026-04-12-test-catalog-skill-bundle-asset-enrichment.md`**.

### Bundle directories outside the package (relative and absolute sources)

Authors can nominate a bundle directory with a **relative path from the package working
directory** (including **`../…`** to a peer or parent folder) or an **absolute** path.
Resolution matches **`atp package … component add`**: the CLI resolves the directory the
same way it does for components, then stages a copy into **`stage.tar`**.

- **Manifest `path`**: when the bundle directory is **inside** the package root, **`path`**
  is the usual **relative** path under that root (forward slashes in YAML). When the
  directory is **outside** the package, **`path`** is the source directory **basename**
  only, so archive prefixes stay flat and predictable (analogous to external components).

- **Legacy** (`atp package bundle add …`): in-package bundles keep the existing tarball
  layout behaviour; **external** trees are staged with the same basename prefix.

- **Multi** (`atp package part <n> bundle add …`): the same rules apply per part; bundle
  **basename uniqueness** across the whole package still applies at add time.

- **`atp package bundle remove`** and **`atp package part <n> bundle remove`** accept the
  same path forms when removing by filesystem location; removal keys off the manifest and
  archive and **does not require** the original source directory to still exist on disk.

Implementation and test pointers: **`docs/notes/2026-04-13-coding-bundle-relative-source-paths.md`**
and **`docs/notes/2026-04-13-test-bundle-relative-source-paths.md`**.

## Validation, summary, and catalog

- `atp validate package` and `atp package summary` choose Multi versus legacy rules from
  the resolved manifest shape.

- `atp catalog add package` runs validation first. Catalog asset entries use the same
  `part_N_Type/` prefixes as the tarball so extraction and install stay aligned.

- **`atp catalog remove <package-name>`** drops a **user** catalog package: it removes the
  matching object from **`packages.user`**, rewrites **`atp-catalog.yaml`**, and deletes
  **`user_packages/<package-name>/`** under the Station when present. Standard-only names
  are rejected with exit **1** and stderr guidance (this subcommand does not edit
  **`packages.standard`**). Unless **`--from-catalog-only`** is passed, the command also
  refuses when any path in **`atp-safehouse-list.yaml`** has a manifest that still lists the
  package; **`--and-from-projects`** removes from those Safehouses first (so agent cleanup
  can still resolve the catalog entry), then performs the catalog drop.

## Install, removal, and listing

- The Safehouse manifest can record whether a row came from a Multi or legacy catalog
  layout (`install_layout`). That flag is metadata for tooling; removal semantics are
  unchanged.

- `atp install` walks all parts for Multi packages. If asset copy fails mid-way, rollback
  is best-effort for files recorded during that attempt; dependency recursion and every
  binary edge case are not guaranteed to roll back as a single transaction.

- **`atp install`** and **`atp remove safehouse`** need a non-empty Safehouse **`agent`**
  (assign with **`atp agent <name>`** after **`safehouse init`**). Unset or blank
  **`agent`** fails fast with stderr guidance instead of assuming **cursor**.

  Implementation and test pointers:
  **`docs/notes/2026-04-13-coding-safehouse-agent-required-install-remove.md`**,
  **`docs/notes/2026-04-13-test-safehouse-agent-required-for-install-remove.md`**.

- `atp remove safehouse` and `atp remove station` remove the entire **installed** package
  by name from the project Safehouse or from the Station install layout, respectively.
  There is still no CLI to drop individual parts from an installed package.

- **`atp catalog remove <package-name>`** removes the package from the **Station catalog
  index** (user list + **`user_packages/`** materialisation). It is not the same as
  **`remove safehouse`** or **`remove station`**, which operate on install manifests and
  copied agent files. The default guard only sees Safehouses registered in
  **`atp-safehouse-list.yaml`** (typically via **`atp safehouse init`**); **`--and-from-projects`**
  chains **`remove safehouse`**-style cleanup for those projects before deleting the user
  catalog row.

- `atp catalog list --verbose`, `atp station list --extended`, and
  `atp safehouse list --extended` add short type summaries (Multi lists part kinds in
  brackets). Some YAML load failures on those code paths exit 2.

# Per-agent installation

## Intent and scope

ATP treats each agent product (Cursor, Claude Code, Gemini CLI, Codex CLI, and related
Station ids) as having its own on-disk contract: where rules and skills live, whether MCP
is a standalone JSON file or a slice of `settings.json` or TOML tables, and how hooks are
represented. Support varies by part kind and agent: some combinations are fully wired,
some are partial (for example prompts that mostly live in chat or MCP rather than a single
drop directory), and experimental surfaces stay explicitly open until products stabilise.

MCP and hook installs often merge into existing host files. When a server name or hook
dedupe key already exists but the JSON body differs, install stops unless the user opts
into replacing the entry or skipping structured config merges for that run.

## Pipeline and agent selection

- Staged part handles (`StagedPartInstallInput`, part kind, helpers that group extracted
  files by `part_N_Type/` or synthesise a single legacy part) sit between the catalog
  manifest and provider code.

- `InstallContext` carries the normalised agent id, install layer (project versus user
  scope as implemented), project root, layer root (for example `.cursor/` or `.gemini/`),
  and the staging directory for the extracted package.

- `atp agent` and handover persist only agent ids that have a provider implementation;
  unknown ids are rejected with a clear error.

- Catalog install and Safehouse remove paths read the same Safehouse **`agent`** field;
  there is no silent default to **cursor** when it is unset.

## Agent behaviour in this release (project install)

#### Cursor

Project rules (including optional `.mdc` assembly), prompts, skills (shared planner),
hooks (`hooks.json` plus scripts under `hooks/`), and MCP (`mcp.json`). Command and
Experimental markdown components follow the same markdown destinations as other
rule-like material; bundle binaries still install after the provider pass.

#### Gemini CLI

Materialises under the project `.gemini/` tree: rules, prompts, skills, TOML custom
commands, hook scripts, and both `mcpServers` and `hooks` merged into `.gemini/settings.json`
when merges are enabled.

#### Codex CLI

Skills under `.agents/skills/` at repo root, MCP as `[mcp_servers.*]` inside
`.codex/config.toml`, hooks in `.codex/hooks.json` when the hooks feature is enabled. The
installer can turn on `codex_hooks` in config when hook parts need Codex to read hooks; if
the file already disables hooks, install surfaces a conflict unless forced.

#### Claude Code

Project installs target `.claude/`: rule-like markdown, skills, MCP merged into
repo-root `.mcp.json` (or user-layer equivalents when Station user paths are in play),
and hooks merged into `settings.json` beside other Claude JSON settings, following the
same merge and provenance rules as other providers.

Shared skill planning keeps `SKILL.md` validation, optional sibling files, and bundle
placeholder patching consistent across agents that use the Agent Skills layout.

## Operator controls

- Ambiguous MCP or hooks merges print one line naming the conflict, the target file, and
  whether to use force or skip. With `--verbose` or when `DEBUG` contains `atp`, an extra
  single-line JSON payload can follow for automation.

- `--force-config` replaces the conflicting fragment with the package version.
  `--skip-config` leaves existing JSON or TOML merge targets untouched for that install
  while still copying tree assets where policy allows. The two flags are mutually
  exclusive.

## Recursive payloads and rollback

Hook script trees and extra skill files may copy as whole directories when the install
plan marks them recursive. Merge helpers append journal entries so Safehouse removal can
roll back MCP, hooks, and Codex feature-flag fragments without deleting unrelated user
content wholesale.

Implementation and test pointers for Codex **`[features].codex_hooks`**, journal
**`codex_features`**, and recursive directory **`raw_file_copy`**:
**`docs/notes/2026-04-11-coding-note-codex-hooks-recursive-copy-journal-refactor.md`**,
**`docs/notes/2026-04-11-test-note-codex-hooks-journal-tree-copy.md`**.

## Documented non-goals

Enterprise-only rule sources, team hook tiers, full global-directory parity, and similar
items are explicitly out of scope for this step; user-global paths exist in Station
config but behaviour is not guaranteed to match project installs in every combination.

# Package manifest rules (0.2.3 schema)

The root `Package` record still carries name, version, developer, license, copyright, and
`type`. From 0.2.3 onward the default type is Multi. When `type` is Multi, `parts` must
contain at least one part object before the package validates; root `usage`, `components`,
and `bundles` are not required because each part carries its own usage and optional
components and bundles.

When `type` is any single canonical part kind other than Multi (Rule, Skill, Mcp, and so
on), the file follows legacy rules: root
`usage`, `components`, and optional `bundles` are required, `parts` must not be populated,
and validation uses the legacy algorithm. Mixing populated `parts` with a non-Multi root
type is invalid.

Each part object has mandatory `type` and `usage`, optional `components` and `bundles`, and
the same bundle object shape as before: `path`, optional `exec-filter`, or `skip-exec: true`
when the bundle has no programs. Repeating the same part
`type` on multiple parts is valid; ATP does not emit stderr warnings solely for that
pattern. Authoring-time edits to parts stay in the
package working tree; after catalog add or install, only whole-package remove applies.

Catalog install with Mcp or Hook parts may merge into agent-specific files as described in
the installer section above; exact paths depend on the Safehouse agent and Station
`agent-paths` entries.

# Contributing new or changed AgentProvider code

Contributors implement the `AgentProvider` interface: `planInstall` builds an ordered
`ProviderPlan` for one staged part without touching disk; `applyPlan` executes that plan;
`planRemove` builds removal steps from provenance recorded during install. Merge options
(`forceConfig`, `skipConfig`) flow from `atp install` into helpers that touch JSON or
TOML.

Actions are discriminated values (markdown writes, raw file copies, MCP JSON merge, Codex
TOML MCP merge, hooks merge, deletes, and so on). Each action carries provenance so
uninstall can remove only ATP-owned fragments.

MCP and hooks merges should go through the shared executor or the same merge helpers with
`mergeConfigTargetLabel` built from the layer root and relative path; otherwise ambiguity
errors fall back to a generic “merged configuration file” label.

Wiring a new agent touches normalisation and supported-id lists, routing gates that decide
when the provider runs, the catalog install branch that constructs the provider and runs
plan then apply, removal destination helpers, and Safehouse teardown that replays journal
fragments and skill bundle trees. New work should add unit tests on plan shape and
integration tests that drive the built CLI against a temp Station and Safehouse.
